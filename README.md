## Token Optimization

We're obsessed with efficiency. Your credits go further.

- **Hierarchical Memory** â€” 52% token reduction through smart consolidation
- **Semantic Compression** â€” Compress context without losing meaning
- **Query Classification** â€” Simple questions use cheaper models

- **MD File Progressive Disclosure** â€” 4-level disclosure for .bos.md files

### .md Progressive Disclosure Levels

| Level | Content | Tokens |
|-------|---------|--------|
| 0 | Index only (filename + description) | ~10/file |
| 1 | Headers + summary | ~50-100/file |
| 2 | Key sections expanded | ~200-500/file |
| 3 | Full content | All tokens |

**70%+ of queries answered without full file load.**

**Target:** 30-50% cost savings without quality loss.

## Data Architecture

**Dual-Database System:** Vector + Structured working together.

| Layer | Technology | Purpose |
|-------|------------|---------|
| **Structured DB** | PostgreSQL | Users, businesses, tasks, stages, relationships |
| **Vector DB** | pgvector | Semantic search, memory retrieval, context matching |
| **File Context** | .bos.md files | User-editable business context |
| **Cache** | Redis (optional) | Session state, hot data |

### Why Both?

```
User asks: "What did we decide about pricing last month?"

Structured DB: Find conversations from last month
Vector DB: Semantic search for "pricing decisions"
Combined: Precise + Relevant results
```

**Vector DB Powers:**
- Semantic memory retrieval
- Similar context finding
- Relevance scoring for progressive disclosure
- Agent knowledge matching

**Structured DB Powers:**
- User accounts, permissions
- Business stages, relationships
- Task tracking, milestones
- Analytics, metrics history
